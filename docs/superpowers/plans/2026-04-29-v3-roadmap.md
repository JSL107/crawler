## 1. SOTA Foundations (#10) — 모든 V3 의 전제

### 1.1 AST 기반 구조적 RAG (Code Graph)

**문제**: BE-3 lite 의 LLM 텍스트 분석으로는 "이 Port 를 구현하는 Adapter" 같은 multi-hop 추론 불가. BE-1 의 RCA 도 호출 그래프 없으면 부정확.

**도입 단계**:
1. **PoC**: Tree-sitter Node binding 으로 `src/` 의 모든 `.ts` 파일을 함수/클래스 단위 chunk. 각 chunk 의 imports / extends / implements / call 을 그래프 인덱스로 추출.
2. **저장소 선택**: (a) 인메모리 + JSON snapshot, (b) SQLite, (c) Neo4j. 시작은 (a) — 이대리는 단일 사용자라 hot reload 만 잘 되면 충분.
3. **Agent 통합**: `code-graph-query.usecase` — "Port X 구현 Adapter / 함수 Y 호출자 / 클래스 Z 상속 트리". BE-3 + BE-1 + BE-2 가 공통 사용.

**비용 추정**:
- 라이브러리: `tree-sitter`, `tree-sitter-typescript` (런타임 free)
- 저장소: 인메모리 0원 / Neo4j Aura free tier 가능
- 구현 시간: 1주 (PoC) + 2주 (Agent 통합)

**리스크**: chunk 정확도. 기존 `agent_run.output` FTS 와 보완 관계.

### 1.2 안전한 격리 실행 환경 (Sandbox)

**문제**: BE-1/BE-2/BE-4 가 `pnpm test` / git checkout / patch 적용 등을 메인 호스트에서 실행 → 시크릿 노출, 의도치 않은 파일 변경 위험.

**도입 단계**:
1. **MVP**: Docker container 일회성. `docker run --rm --network none -v $REPO:/repo node:20 /bin/bash -c "pnpm install && pnpm test"`. ephemeral.
2. **강화**: Firecracker MicroVM (AWS Fargate / fly.io) — 프로세스 격리 한 단계 위.
3. **망 분리**: `--network none` 또는 egress allowlist (GitHub API 만).

**비용 추정**:
- Docker: 로컬 0원, 운영 시 호스팅 비용
- Firecracker: AWS Fargate $40/월 추정 (소규모)
- 구현 시간: 2주 (Docker MVP) + 4주 (Firecracker)

**리스크**: 기존 CLI provider 의 throwaway HOME 패턴과 충돌 회피 — sandbox 안에서도 동일 격리 적용 필요.

### 1.3 MCP + RBAC

**문제**: BE-4 의 git push / PR write / DB 변경 등 외부 부작용은 단일 권한 토큰으로는 위험. Confused Deputy 가능성.

**도입 단계**:
1. **MCP 서버 도입**: GitHub / Slack / DB / 쉘 명령을 MCP server 로 캡슐화.
2. **RBAC**: 작업 단위 token. BE-4 가 "PR 수정" 만 할 수 있고 "issue close" 는 못 함.
3. **감사 로그**: 모든 MCP 호출을 `agent_run.evidence_record` 에 기록.

**비용 추정**:
- MCP: TypeScript SDK 사용 (`@modelcontextprotocol/sdk`)
- 구현 시간: 3주

## 2. BE Agent 별 단계 plan

### #6 BE-3 영향도 분석 강화 (현재 lite → full)

**전제**: SOTA Foundation 1.1 (Code Graph) 1단계 PoC.

**단계**:
1. `code-graph-query` 로 schema.prisma 의 model 변경이 영향 미치는 src 파일 식별
2. `prisma generate --dry-run` 동등 결과 시뮬레이션 (TS 타입 영향)
3. SchemaProposal 에 `affectedFiles: string[]` 필드 추가, formatter 가 노출
4. 기존 spec 확장 (lite 5케이스 → full 8케이스)

**예상 commit 수**: 5 ~ 7

### #7 BE-1 Auto-SRE

**전제**: SOTA Foundation 1.2 (Sandbox) MVP.

**단계**:
1. **Webhook 수신**: Datadog/Sentry → POST `/v1/agent/sre`. HMAC + payload 정규화.
2. **Stack trace 파싱**: 기존 코드와 trace 매칭 → 영향 함수 식별 (Code Graph 활용).
3. **Sandbox 재현**: 에러 trigger 하는 임시 spec 작성 → sandbox 에서 실행해 reproduce 확인.
4. **Fix 작성**: LLM 이 patch 제안 → sandbox 에서 `pnpm test` 통과 확인.
5. **PR Draft**: GitHub API 로 fix branch + PR draft 생성 (자동 merge X — HITL).
6. **Slack 알림**: `[✅ Fix PR Draft 검토]` 버튼.

**예상 commit 수**: 12 ~ 18

### #8 BE-2 AST Test Gen

**전제**: SOTA Foundation 1.1 (Code Graph) — Tree-sitter 만으로 충분.

**단계**:
1. `/be-test <파일>` 슬래시
2. Tree-sitter 로 대상 파일 AST 추출 → 분기(`if`/`switch`/`?:`) 모든 path 식별
3. 주입된 Port 인터페이스 → 자동 Mock 생성 (Jest)
4. spec 작성 → sandbox 에서 실행 → lint/type 에러 시 self-correction 루프 (max 3회)
5. 완성된 `*.spec.ts` 반환

**예상 commit 수**: 10 ~ 15

### #9 BE-4 Closed-Loop Auto-Remediation

**전제**: SOTA Foundation 1.3 (MCP RBAC) — GitHub PR write 권한 격리.

**단계**:
1. PR webhook 수신 → diff 파싱
2. Code Graph + LLM 으로 사소한 컨벤션 위반 식별 (`if {}` 누락 / 매직 넘버 / 변수명 줄임말)
3. Sandbox 에서 fix 적용 → `pnpm lint:check && pnpm test` 통과 확인
4. 임시 브랜치 push (`fix/agent-review-<PR번호>`)
5. Slack 알림 + `[✅ 이 fix 로 PR 덮어쓰기]` 인터랙티브 버튼 (HITL)
6. 사용자 클릭 시 GitHub API 로 force-push 또는 merge

**예상 commit 수**: 15 ~ 20

**리스크**: ★★★ 가장 큼 — git push 자동화는 사용자 신뢰 깨질 가능성. HITL 디자인 신중.

### #11 PoExpandApplier (Stage 2)

**전제**: 없음 (PreviewGate + ApplyPreview 이미 구현됨).

**단계**:
1. `src/agent/po-expand/infrastructure/po-expand.applier.ts` 작성
2. apply 시 LLM 으로 outline → full PRD 변환 (별도 system prompt)
3. PreviewGate 등록 (`app.module.ts:62 appliers` 배열)
4. `apply-preview.usecase.spec.ts` 의 PO_EXPAND 케이스 활성화
5. 슬래시 결과에 `[🚀 Stage 2 — full PRD 생성]` 버튼

**또는 deprecate**:
1. `PREVIEW_KIND.PO_EXPAND` enum 제거
2. spec 의 PO_EXPAND 케이스 삭제
3. 관련 주석 정리

**예상 commit 수**: 5 (구현) / 2 (deprecate)
